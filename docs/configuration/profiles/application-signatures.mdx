# Application Signatures

Manage how SafeSquid identifies and categorizes different applications and websites that people use on your network.

## Global Settings

Configure section-wide settings that apply to all entries.

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| **Enabled** | `BOOL` | `app_enabled` | Turn on or off the use of Application Signatures (to identify and categorize applications). |


## Configuration Fields

The following fields are available when creating rules in this section.

### Enabled

**Type:** `BOOL`  
**Default:** `enabled`  



Turn on or off this rule.


**Available Options:**

| Option | Value | Description |
|--------|-------|-------------|
| false | `false` | Turn off this rule. |
| true | `true` | Turn on this rule. |


### Comment

**Type:** `MULTILINE_STRING`  




Add notes here to explain what this rule does and why you created it. This helps you and others understand it later.


### Application signatures

**Type:** `STRING_LIST`  
**Default:** `add_app_category,add_app_signatures`  

**Supports Negation:** Yes (prefix with `!` to exclude)  

Comma separated list of Request Profiles to which this entry will be applied. This could be one or more Request Profiles already applied(ADDED REQUEST PROFILES) to the connection, due to this entry in the list. If kept Blank, it will apply to all connections irrespective of any applied request profile.


### Method

**Type:** `STRING_SELECT_MANY`  
**Default:** `method`  



This rule applies to requests matching the selected method types (like GET, POST). If you want this rule to apply to all requests regardless of method, don't select any method.


**Available Options:**

| Option | Value | Description |
|--------|-------|-------------|
| GET | `GET` | GET method is used to retrieve information from server using a given requested web-server. Requests using GET should only retrieve data and should have no other effect on the data. Select this if you want to apply this entry for requests with GET method. |
| POST | `POST` | POST method is used to submit data to be processed (Example: customer information, file upload etc. using HTML forms). The data is included in the body of the request. This may result in the creation of a new resource or the updates of existing resources or both. Select this if you want to apply this entry for requests with POST method. |
| CONNECT | `CONNECT` | CONNECT Converts the request connection to a transparent TCP/IP tunnel, usually to facilitate SSL-encrypted communication(HTTPS) through an unencrypted HTTP proxy(e.g. SSL tunnelling). Select this if you want to apply this entry for requests with CONNECT method. |
| HEAD | `HEAD` | HEAD method is identical to GET. However in response to HEAD request server shall not return a message body. Select this if you want to apply this entry for requests with HEAD method. |
| PUT | `PUT` | PUT method is used to upload a representation of the specified resource. If request for any web-server refers to an already existing resource, it will be modified. Otherwise server can create the resource with present URI(Uniform Resource Identifier). Select this if you want to apply this entry for requests with PUT method. Unused. |
| DELETE | `DELETE` | DELETE method removes the specified resource. It intends to delete the resource or move it to an inaccessible location identified by the requested web-server. Select this if you want to apply this entry for requests with DELETE method. Unused. |
| TRACE | `TRACE` | TRACE Echoes back the received request, so that a client can see what intermediate servers are adding or changing in the request. The final recipient of the request SHOULD reflect the message received back to the client as the entity-body of a 200 (OK) response. Select this if you want to apply this entry for requests with TRACE method. Unused. |
| OPTIONS | `OPTIONS` | OPTIONS method returns the HTTP methods that server supports for specified URL. OPTIONS can be used to check the functionality of a web server by requesting \* instead of a specific resource. Select this if you want to apply this entry for requests with OPTIONS method. Unused. |


### Protocol

**Type:** `STRING_SELECT_MANY`  
**Default:** `protocol`  



This rule applies to requests matching the selected connection types (like HTTP, HTTPS). If you want this rule to apply to all requests regardless of connection type, don't select any protocol.


**Available Options:**

| Option | Value | Description |
|--------|-------|-------------|
| FTP | `FTP` | File Transfer Protocol(FTP) is a standard network protocol used to transfer files from any host to another host over a TCP-based network. Select this if you want to apply this entry for FTP clients. |
| HTTP | `HTTP` | Hypertext Transfer Protocol(HTTP) is an application protocol for distributed, collaborative, hypermedia information systems. HTTP is the protocol to exchange or transfer hypertext. Select this if you want to apply this entry for HTTP clients. |
| HTTPS | `HTTPS` | Hypertext Transfer Protocol Secure (HTTPS) is a communications protocol for secure communication over a computer network. **Very first request for HTTPS will be CONNECT request, subsequently it will be changed to HTTPS. Therefore lower down in protocol it is HTTPS request. Select this if you want to apply this entry for HTTP clients who want to access websites using SSL.** |


### Mime type

**Type:** `STRING`  




This rule applies to requests matching specific file types (MIME types) that are being sent. File types identify what kind of content is being transmitted (like images, documents, etc.). Example: application/x-www-form-urlencoded. If you want this rule to apply to all requests regardless of file type, leave it blank.


### Port range list

**Type:** `PORT_RANGE_LIST`  




Comma separated list of ports or port ranges to which this entry will be applied. Here portrange indicates TCP port(s) number on which the server is listening. Example: value "80,21-25" means port 80 and port range from 21 to 25. Applies to all ports if left blank.


### URL Command

**Type:** `STRING_LIST`  
**Default:** `url_commands`  



Comma separated list of URL command(s) to which this entry will be applied. If this entry should be applied to all requests irrespective of any urlcommand leave it blank.


### Minimum Post Data Size

**Type:** `FILE_SIZE`  
**Default:** `0`  
**Optional:** Yes  


This entry applies only if Request body is more than this size. Example: Content-Length: 48


### Maximum Post Data Size

**Type:** `FILE_SIZE`  
**Default:** `0`  
**Optional:** Yes  


This entry applies only if Request body is less than this size. Example: Content-Length: 480


### File

**Type:** `STRING`  




This rule applies to the file path part of a web address (the part after the website name in the URL). Example: (cgi-bin|\\?) will match queries in a URL. Leave it blank to apply this rule to all requests regardless of the file path.


### Registered Domain

**Type:** `STRING`  




Regular expression matching the Registered domain name of the requested host.  
Example:  
Registered Domain: **^wikipedia\\.** will match all sites like wikipedia.org, www.wikipedia.com, en.wikipedia.org, etc.  
Leave it blank to apply all the requests irrespective of registered domain.


### Host Name

**Type:** `STRING`  




Regular expression matching the FQDN (Fully Qualified Domain Name) of the requested host.  
Example:  
Host Name: **mail\\.** will match all sites like mail.google.com, www.email.com, etc.  
Leave it blank to apply all the requests irrespective of host name.


### Referer Domain Name

**Type:** `STRING`  




Regular expression matching the registered domain of the referrer host.  
Example:  
Referer Domain Name: **^wikipedia\\.** will match all requests with referer hostnames like wikipedia.org, www.wikipedia.com, en.wikipedia.org, etc.  
Leave it blank to apply all the requests irrespective of referer domain.


### Referer Host Name

**Type:** `STRING`  




Regular expression matching the FQDN (Fully Qualified Domain Name) referrer host.  
Example:  
Host Name: **mail\\.** will match all requests referred by sites like mail.google.com, www.email.com, etc.  
Leave it blank to apply all the requests irrespective of host name.


### Referer

**Type:** `STRING`  




This entry applies to regular expression matching the referer(the address of the resource from which the requested URI was obtained) of requested web-server. Example: Referer: http://en.wikipedia.org/wiki/Main\_Page Leave it blank to apply all requests irrespective of referer of header.


### User Agent

**Type:** `STRING`  




This entry applies to regular expression matching to a specific internet client or web client(application requesting the information from the server). Example: User-Agent: Mozilla/5.0 (X11; Linux x86\_64; rv:12.0) Gecko/20100101 Firefox/21.0 Leave it blank to apply all requests irrespective of any application requesting the information from the server.


### X-Forwarded-For

**Type:** `STRING`  




This entry applies to regular expression matching the X-Forwarded-For(the IP address from where the original request came from either proxy or load balancer). Example: X-Forwarded-For: 129.78.138.66, client1, proxy1 . To apply all requests irrespective of X-forwarded-for leave it blank.


### Request header pattern

**Type:** `STRING`  




This entry applies to regular expression matching the request header pattern. Apply to all request patterns if left blank.


### Added application signatures

**Type:** `STRING_LIST`  
**Default:** `add_app_signatures`  



Comma separated list of Request Profiles that will be added to connection, if the above specified tests result true.


### Removed application signatures

**Type:** `STRING_LIST`  
**Default:** `add_app_signatures`  



Comma separated list of Request Profiles to be removed when all the above rules match. If any of these Request profiles have been already applied to the connection by any of the other Request Profile rules, they will be removed.


### category

**Type:** `STRING_LIST`  
**Default:** `add_app_category,`  



Category of application signatures

## Subsections

This section contains the following subsections for organizing rules:

### Application Signatures List

The following rules are tested for each connection. The testing is done in top-down order. The first entry that matches the Request Profile of a connection, will be applied to it. Click on Add below, to add a new entry.

The following rules are tested for each connection. The testing is done in top-down order. The first entry that matches the Request Profile of a connection, will be applied to it. Click on Add below, to add a new entry.
