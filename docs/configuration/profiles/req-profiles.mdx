# Request Types

Use Request Profiler to organize and categorize requests based on what people are trying to access.

This section lets you create rules based on how the request is made, what type of connection it is, where it came from, and what browser or app is being used.

This section lets you block or allow specific websites by their name.

You can use this section to make social media sites (like Facebook and Twitter) read-only, so people can view content but cannot like, comment, share, chat, or play games.

This section helps you turn on safe search for Google, Yahoo, and YouTube

## Global Settings

Configure section-wide settings that apply to all entries.

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| **Enabled** | `BOOL` | `req_policy` | Turn on or off request types (categorizing what people are trying to access). |


## Configuration Fields

The following fields are available when creating rules in this section.

### Enabled

**Type:** `BOOL`  
**Default:** `enabled`  



Turn on or off this rule.


**Available Options:**

| Option | Value | Description |
|--------|-------|-------------|
| FALSE | `FALSE` | Disable this entry. |
| TRUE | `TRUE` | Enable this entry. |


### Comment

**Type:** `MULTILINE_STRING`  
**Default:** `This entry ...`  



For documentation, and future references, explain the relevance of this entry with your policies.

That is, by reading the policies, a future user can understand the purpose of that entry.


### Trace Entry

**Type:** `STRING_SELECT_ONE`  
**Default:** `profile_tracing`  



Enable or Disable tracing of this entry

Select "Yes" to debug the application of entry using SafeSquid logs.

Enabling entry tracing, is useful if you wish to validate, its application.


**Available Options:**

| Option | Value | Description |
|--------|-------|-------------|
| FALSE | `FALSE` | Select this option to disable profile tracing. |
| TRUE | `FALSE` | Select this option to enable profile tracing. |


### Request Types

**Type:** `STRING_LIST`  
**Default:** `add_app_category,add_app_signatures,add_req_types,int_req_types`  

**Supports Negation:** Yes (prefix with `!` to exclude)  

Comma separated list of Request Profiles to which this entry will be applied. Due to this entry in the list, one or more Request Profiles are already applied (ADDED REQUEST PROFILES) to the connection. If kept Blank, it will apply to all connections irrespective of any applied request profile.


### Method

**Type:** `STRING_SELECT_MANY`  
**Default:** `method`  



This entry applies to requests matching the selected list of Method(s). If this entry should be applied to all requests irrespective of methods, do not select any method.


**Available Options:**

| Option | Value | Description |
|--------|-------|-------------|
| GET | `FALSE` | GET method is used to retrieve information from server using a given requested web-server. Requests using GET should only retrieve data and should have no other effect on the data. Select this if you want to apply this entry for requests with GET method. |
| POST | `FALSE` | POST method is used to submit data to be processed (Example: customer information, file upload etc. using HTML forms).  
The data is included in the body of the request. This may result in the creation of a new resource or the updates of existing resources or both. Select this if you want to apply this entry for requests with POST method. |
| CONNECT | `FALSE` | CONNECT Converts the request connection to a transparent TCP/IP tunnel, usually to facilitate SSL-encrypted communication(HTTPS) through an unencrypted HTTP proxy(e.g. SSL tunnelling). Select this if you want to apply this entry for requests with CONNECT method. |
| HEAD | `FALSE` | HEAD method is identical to GET. However in response to HEAD request server shall not return a message body. Select this if you want to apply this entry for requests with HEAD method. |
| PUT | `FALSE` | PUT method is used to upload a representation of the specified resource. If request for any web-server refers to an already existing resource, it will be modified. Otherwise server can create the resource with present URI(Uniform Resource Identifier). Select this if you want to apply this entry for requests with PUT method. |
| DELETE | `FALSE` | DELETE method removes the specified resource. It intends to delete the resource or move it to an inaccessible location identified by the requested web-server. Select this if you want to apply this entry for requests with DELETE method. |
| TRACE | `FALSE` | TRACE Echoes back the received request, so that a client can see what intermediate servers are adding or changing in the request. The final recipient of the request SHOULD reflect the message received back to the client as the entity-body of a 200 (OK) response. Select this if you want to apply this entry for requests with TRACE method. |
| OPTIONS | `FALSE` | OPTIONS method returns the HTTP methods that server supports for specified URL. OPTIONS can be used to check the functionality of a web server by requesting \* instead of a specific resource. Select this if you want to apply this entry for requests with OPTIONS method. Unused. |


### Protocol

**Type:** `STRING_SELECT_MANY`  
**Default:** `protocol`  



This entry applies to requests matching the selected list of protocol(s).

If this entry should be applied to all requests irrespective of protocols, do not select any protocol.


**Available Options:**

| Option | Value | Description |
|--------|-------|-------------|
| FTP | `FALSE` | File Transfer Protocol(FTP) is a standard network protocol used to transfer files from any host to another host over a TCP-based network. Select this if you want to apply this entry for FTP clients. |
| HTTP | `FALSE` | Hypertext Transfer Protocol(HTTP) is an application protocol for distributed, collaborative, hypermedia information systems. HTTP is the protocol to exchange or transfer hypertext. Select this if you want to apply this entry for HTTP clients. |
| HTTPS | `FALSE` | Hypertext Transfer Protocol Secure (HTTPS) is a communications protocol for secure communication over a computer network. Very first request for HTTPS will be CONNECT request, subsequently it will be changed to HTTPS. Therefore lower down in protocol it is HTTPS request. Select this if you want to apply this entry for HTTP clients who want to access websites using SSL. |


### Content type

**Type:** `STRING_LIST`  
**Default:** `add_content_signatures`  

**Supports Negation:** Yes (prefix with `!` to exclude)  

This entry applies to regular expression matching to the MIME-type.

Here MIME-type indicates the entity-body sent to the recipient (used with POST and PUT requests).

Request Type is the type of request sent by the requester such as an incident or request for any information.

Example: Content-Type: application/x-www-form-url encoded. If this entry should be applied to all requests irrespective of MIME-type, leave it blank.


### Port range list

**Type:** `PORT_RANGE_LIST`  




Comma separated list of port or port ranges to

which this entry will be applied.

Here port range indicates TCP port(s) number on which the server is listening.

Example: value "80, 21-25" means port 80 and port range from 21 to 25.

It applies to all ports if left blank.


### URL Command

**Type:** `STRING_LIST`  
**Default:** `url_commands`  



Comma separated list of URL command(s) to which this entry will be applied.

If this entry should be applied to all requests irrespective of any url command, leave it blank.

https

For https requests.

!https

For other than https requests.


### Minimum Post Data Size

**Type:** `LONG_FILE_SIZE`  
**Default:** `0`  



This entry applies only if Request body is more than this size.  
Example: Content-Length: 48


### Maximum Post Data Size

**Type:** `LONG_FILE_SIZE`  
**Default:** `0`  



This entry applies only if Request body is less than this size.  
Example: Content-Length: 480


### File

**Type:** `STRING_LIST`  
**Default:** `add_file_extensions`  



This entry applies to regular expression matching the file portion of requested web-server (the part of a URL that succeeds hostname).

Leave it blank to apply all the requests irrespective of file portion of URI (Uniform Resource Identifier).

  
Example: (cgi-bin|\\?) will apply to queries in a URL.


### Host Name

**Type:** `STRING`  




This entry applies to regular expression matching the hostname(website) of requested web-server.

Leave it blank to apply this entry to all the requests irrespective of hostname(s).

  
Example: Host: en.wikipedia.org


### Smart TLD

**Type:** `BOOL`  
**Default:** `smart_tld`  



Enable this option if the regex for Host Name should be used to match irrespective of the TLD.  
Specify google$ in Host Name and set smart TLD to enabled to ensure match for google.com or google.co.uk or google.co.in.  
If you strictly want to create a policy for mail.google.com, then you should set Smart TLD to disabled and mention Host Name as mail\\.google\\.com


**Available Options:**

| Option | Value | Description |
|--------|-------|-------------|
| FALSE | `TRUE` | The regex contains specific TLD for which the match is required |
| TRUE | `FALSE` | Ignore TLD of the requested Host Name while matching regex. |


### Referer

**Type:** `STRING`  




This entry applies to regular expression matching the referrer (the address of the resource from which the requested URI was obtained) of requested web-server.

Leave it blank to apply all requests irrespective of referrer of the header.

  
Example: Referer: http://en.wikipedia.org/wiki/Main\_Page


### User Agent

**Type:** `STRING`  




This entry applies to regular expression matching to a specific internet client or web client (application requesting the information from the server).

Leave it blank to apply all requests irrespective of any application requesting the information from the server.

  
Example: User-Agent: Mozilla/5.0 (X11; Linux x86\_64; rv:12.0) Gecko/20100101 Firefox/21.0


### X-Forwarded-For

**Type:** `STRING`  




This entry applies to regular expression matching the X-Forwarded-For (the IP address from where the original request came from either proxy or load balancer).

Leave it blank to apply for all requests irrespective of X-forwarded.

  
Example: X-Forwarded-For: 129.78.138.66.


### Request header pattern

**Type:** `STRING`  




This entry applies to regular expression matching the request header pattern. Apply to all request patterns, if left blank.


### Added Request Types

**Type:** `STRING_LIST`  
**Default:** `add_app_category,add_app_signatures,add_req_types,int_req_types`  



Comma separated list of Request Profiles that will be added to connection, if the above specified tests results are true.


### Removed Request Types

**Type:** `STRING_LIST`  
**Default:** `add_app_category,add_app_signatures,add_req_types`  



Comma separated list of Request Profiles to be removed when all the above rules match.

If any of these Request profiles have been already applied to the connection by any of the other Request Profile rules, they will be removed.

### Domains and URLs

**Type:** `MULTILINE_STRING`  
**Default:** `This entry ...`  



**List of Domains and URLs.**

Multiple Domains and URLs may be specified.  
This Entry shall be applicable for Requests to any of the specified Domains and URLs.  
**Comma, space, tab, newline** are all acceptable separators (delimiters).

Leaving it blank will cause this Entry to apply to all requests.

**Do not include:**  
The preceeding protocol specifiers such as **http://** or **https://**  
Wild-Card specifiers such as **\***

**Acceptable Examples:**  
safesquid.com  
www.safesquid.com  
safesquid.com/some/file/path


### Match

**Type:** `STRING_SELECT_ONE`  
**Default:** `which`  



Check if the requested host or the referer matches one of the specified domains and URLs

## Subsections

This section contains the following subsections for organizing rules:

### Domains and Urls

Each connection is tested against all Entries from top to bottom for applicability.

Disabled Entries are ignored.

The Added Request Types specified in a matching Entries shall be appended to the Connection's Request Types.

The Removed Request Types specified in a matching Entries shall be removed to the Connection's Request Types.


### Request Types

This is a list of Entries that profiles a request as one or more Request Types.

Each request is tested against the Entries, from top to bottom.

Each Entry is constituted by a set of Rules.

A disabled Entry is ignored.

An Entry is *applicable* if a request satisfies all its Rules.

If an Entry is found to be applicable:

The *Request Type(s)* specified in the **Added Request Types** rule of the Entry, are appended to the request's then list of profiles.

The *Request Type(s)* specified in the **Removed Request Types** rule of the Entry, are removed from the request's then list of profiles.

Thus a request may be labelled with multiple Request Types.
